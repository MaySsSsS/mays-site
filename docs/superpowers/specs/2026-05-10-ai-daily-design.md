# AI Daily Design

## Goal

Add **AI Daily** as a protected MAYS UNIVERSE subpage for reading a daily AI
news digest. The page should feel like a universe branch, not a public tools
module, while still using a simple static-data workflow that fits the current
Next.js site.

## User-Facing Scope

- Replace the homepage `CLASSIFIED` panel with a real `AI DAILY` entrance.
- Route the new section at `/ai-daily`.
- Protect `/ai-daily` and each dated detail page with a page password.
- After authentication, show a reverse-chronological digest index where each
  day is represented by one card.
- Make each card link to a dated detail page such as `/ai-daily/2026-05-10`.
- Keep the card index structurally close to the source blog index: one day per
  card, date-forward browsing, and summary-first scanning.
- Do not show source links in the UI.

## Reading Experience

### Index Page

`/ai-daily` shows the protected digest archive after password entry.

Each daily card includes:

- Date.
- Digest title.
- Generation mode: AI summary or fallback cache.
- A short summary preview.
- A clear link target to the dated detail page.

The page does not include search, saved items, comments, RSS, email delivery,
or multi-source filtering in the first version.

### Detail Page

`/ai-daily/YYYY-MM-DD` shows the full digest for one day after the same password
gate.

The content structure should stay close to the source daily format:

- Today's summary.
- Product and feature updates.
- Research and frontier work.
- Industry, social, or policy impact.
- Open-source projects.
- Social or community highlights.

The exact section names may vary if the source page varies, but the renderer
should preserve the daily-report rhythm rather than compressing everything into
a short newsletter.

## Data Source

The first version uses only the Hex 2077 AI daily blog as the upstream source.
No additional feeds, websites, model release pages, paper feeds, or social
sources are included.

The source is an implementation detail. The UI does not display source links.

## Daily Generation Workflow

Use GitHub Actions for the first version.

The scheduled workflow runs every day at **09:30 Asia/Shanghai** and executes a
Node script that:

1. Derives the source URL for the current date.
2. Fetches the source page.
3. If the source page is unavailable or not yet published, exits without
   creating an empty digest.
4. Parses the page into structured daily content.
5. Calls Zhipu AI to generate the site digest.
6. Writes static JSON data.
7. Commits generated data and lets the normal deployment pipeline publish the
   updated site.

The first version does not retry later in the same day. If the source publishes
after the 09:30 run, the next scheduled run can pick up the next available
digest. A later milestone can add manual dispatch or multiple daily retries if
the actual source update time makes that necessary.

## AI Generation

The primary generation path uses Zhipu Coding Plan-compatible chat completions.

Default runtime configuration:

- `ZHIPU_API_KEY`: required GitHub Secret.
- `ZHIPU_BASE_URL`: defaults to
  `https://open.bigmodel.cn/api/coding/paas/v4`.
- `ZHIPU_MODEL`: defaults to `GLM-5.1`.

The script sends parsed source content to the model and asks for fixed JSON
output. The model must:

- Write Chinese output.
- Keep a section structure similar to the source daily report.
- Summarize and rewrite for personal reading.
- Avoid inventing news or adding facts not present in the source content.
- Return valid JSON with the fields required by the site renderer.

The script must not log API keys, request headers, full source content, or full
model responses.

## Fallback Mode

If the Zhipu call fails, times out, returns invalid JSON, or omits required
fields, the script writes a fallback digest generated from the parsed source
content.

Fallback mode preserves the original page structure as much as practical. It is
allowed to be closer to a mirror/cache than the AI-generated version because
the page is password-protected and intended for personal use.

Each digest records its mode:

- `ai_summary`: generated by Zhipu from parsed source content.
- `mirror_fallback`: generated from parsed source content after AI failure.

The UI may show a small status label for this mode, but should not turn the
fallback state into a distracting error experience.

## Static Data Shape

Generated files live under `public/data/ai-daily/`.

Required files:

- `index.json`: archive index with all dates, latest date, card summaries, and
  generation modes.
- `entries/YYYY-MM-DD.json`: full digest for one date.

Static JSON files may be publicly reachable by direct URL. This is accepted for
the first version. The page password protects the normal reading experience,
not the raw static assets.

The generated JSON must never include:

- `ZHIPU_API_KEY`.
- Authorization headers.
- Raw API request objects.
- Raw model response envelopes.
- Debug logs.

## Password Gate

The first version uses a client-side password gate for the reading UI.

Implementation constraints:

- Store only a password hash in the build environment, not a plaintext
  password.
- Use `AI_DAILY_PASSWORD_HASH` as the build-time environment variable name.
- Compare the user's input against the hash in the browser.
- Store the authenticated state in browser storage so the user does not need to
  re-enter the password for every dated page.
- Treat the gate as a convenience layer, not strong access control, because the
  static JSON files remain public by direct URL.

## Routing And Components

Required route additions:

- `app/ai-daily/page.tsx`: protected archive index.
- `app/ai-daily/[date]/page.tsx`: protected daily detail.

Required component boundaries:

- Password gate component scoped to AI Daily.
- Archive index component that reads `index.json`.
- Daily card component.
- Daily detail renderer for structured sections.
- Small data-loading helpers for static JSON.

Keep implementation aligned with the existing app patterns:

- Next.js App Router.
- TypeScript strict mode.
- CSS Modules.
- No Tailwind or CSS-in-JS.
- No Worker, R2, database, or server-side auth in the first version.

## Homepage Integration

Update the root comic grid so the existing sealed panel becomes:

- Eyebrow: `AI`
- Title: `AI DAILY`
- Link: `/ai-daily`
- Domain label: `maysssss.cn/ai-daily`

The homepage should still read as the MAYS UNIVERSE portal. `AI Daily` is a
peer branch beside Game, Photos, and Tools.

## Secrets And Operations

GitHub Actions needs:

- `ZHIPU_API_KEY`.

Optional workflow variables:

- `ZHIPU_BASE_URL`.
- `ZHIPU_MODEL`.
- `AI_DAILY_PASSWORD_HASH`.

The leaked development key from the planning conversation should be treated as
rotated before production use. The repository should only document the secret
names, never concrete secret values.

## Out Of Scope

- Multi-source aggregation.
- Search.
- Collections or saved articles.
- Public RSS or newsletter output.
- Comments or social features.
- Manual editing admin.
- Worker Cron, R2, KV, or database storage.
- Server-enforced protection for static JSON files.
- Same-day repeated retries.
- Displaying source links in the UI.

## Verification Criteria

The implementation is complete when:

- Homepage links to `/ai-daily` through an `AI DAILY` panel.
- `/ai-daily` shows a password gate before the archive.
- Authenticated users see one card per generated day.
- Each card opens a dated detail route.
- The detail route renders structured full digest content.
- The generation script can create `index.json` and one dated entry from a
  fetched source page.
- The generation script can fall back to `mirror_fallback` when AI generation
  fails.
- No generated file or log contains API secrets.
- `pnpm typecheck`, `pnpm lint`, and `pnpm build` pass.
