# Signal Arena Quant Lab Roadmap

日期：2026-06-12  
状态：v1 主体已实现，本文件作为后续迭代路线图维护
目标路由：`/signal-arena`、`/signal-arena/logs`、`/signal-arena/rank`

## 1. 目标

把当前的 `Signal Arena AI Trader` 改造成一个可学习、可复盘、可迭代的量化模拟盘实验室。

核心变化：

- 新开一个 Signal Arena / Agent World 账号，从 0 开始跑量化模拟盘。
- 取消旧账号作为“AI 决策账号”的继续交易职责；当前公开体验不展示旧账号历史、不做归档入口、不做对照。
- 交易决策由确定性的量化策略产生，不再由 AI 直接决定买卖。
- AI 只作为每周复盘、策略研究和解释辅助，不进入每日下单闭环。
- 网站从“AI 决策页面”升级为“Quant Lab 页面”，展示策略版本、因子分数、风控、收益曲线、交易归因和复盘结论。

这个路线图不是逐行实现计划。`Q-Alpha v1` 主体实现完成后，后续版本仍应继续按阶段拆分 implementation plan。

## 2. 当前系统与目标系统

### 当前系统

当前 Worker 大致流程：

```text
Signal Arena API
  -> top-movers / portfolio / trades / snapshots
  -> deterministic signal-generator
  -> AI prompt / AI decision
  -> risk.ts 风控
  -> trade API
  -> D1 日志 + 公开页面
```

优点：

- 页面、D1 日志、收益曲线、公开 API、风控框架已经存在。
- 已经有 A 股交易时间、T+1、100 股整数倍、可卖数量等基础保护。
- 已经能记录决策过程和前置信号。

问题：

- 最终买卖判断仍依赖 AI，输出有不稳定性。
- AI 对行情的分析依赖 prompt，难以形成可回测、可调参、可复现的策略。
- “为什么买/为什么不买”的解释更像文本推理，不是明确的因子归因。

### 目标系统

目标 Worker 流程：

```text
Signal Arena API
  -> 数据采集与缓存
  -> 指标计算
  -> 策略评分 Q-Alpha v1
  -> 组合构建与风控
  -> trade API
  -> 策略快照 / 因子归因 / 订单结果
  -> 公开页面

每周：
  -> 汇总真实运行和回测结果
  -> AI 复盘生成研究报告
  -> 只提出 v2 假设，不直接下单
```

原则：

- 下单权只属于量化策略和程序风控。
- 每一次策略版本都不可变更；优化时新建版本，例如 `Q-Alpha v2`。
- 所有买卖都能追溯到分数、规则、阈值和当时的数据快照。
- 先做能跑、能解释、能复盘的策略，再做更复杂的机器学习或多智能体研究。

## 3. 第一版策略：Q-Alpha v1

`Q-Alpha v1` 的定位是“多因子趋势动量策略 + 严格风控”。它不是玩具策略，但也不做不可解释的黑箱模型。

### 3.1 标的池

第一版只交易 A 股：

- 以 Signal Arena 的 A 股标的池为准，当前 skill 文档描述为沪深 300 成分股规模的池子。
- 每日或每个交易日开盘前刷新 `stocks-list` / `stocks`，得到可交易 universe。
- 每轮交易不全量深抓所有股票；全量列表低频缓存，候选股和持仓股高频更新。

### 3.2 数据输入

策略需要以下数据：

- `home`：总资产、现金、收益率、排名、市场状态。
- `portfolio`：持仓、市值、成本、可卖数量、盈亏。
- `trades`：近期交易和 pending 订单，用于防重复交易和反复打脸。
- `snapshots`：账户曲线。
- `stocks` / `stocks-list`：A 股候选池。
- `stock-history`：历史价格、成交量、日内或日线数据。
- `top-movers`：短线热度参考，只作为候选补充，不作为核心买入理由。

第一版不假设有完整基本面数据。如果未来 Signal Arena 或外部数据源能稳定提供财务指标，再新增质量因子。

### 3.3 指标

第一版指标全部可解释：

- 趋势：`MA20 > MA60`、现价高于 `MA20`、均线斜率。
- 动量：`5D / 10D / 20D` 收益率。
- 突破：收盘价接近或突破 `20D` 高点。
- 波动：`20D` 日收益波动率，波动过高扣分。
- 回撤：距离近期高点的回撤幅度。
- 成交量：成交量相对均量放大，但不追极端脉冲。
- 持仓风险：单只仓位、浮盈浮亏、是否可卖。
- 行为约束：近期刚卖出的标的降低买入优先级，避免来回交易。

### 3.4 评分

建议第一版总分为 `0-100`：

```text
score =
  30% 趋势分
+ 25% 动量分
+ 15% 突破分
+ 10% 成交量确认
+ 10% 组合适配分
- 10% 波动惩罚
- 10% 追高惩罚
- 10% 近期交易惩罚
```

分数不是绝对真理，只是策略的一致语言。后续 v2 / v3 的优化应当围绕“哪项因子贡献了收益，哪项造成回撤”展开，而不是凭感觉改阈值。

### 3.5 买入规则

第一版买入规则：

- 候选股 `score >= 70` 才能进入买入候选。
- 每轮最多买入 1 只。
- 单日最多新增 1-2 个买入动作，具体由实现阶段根据结算节奏确定。
- 买入后单股仓位不超过总资产 `20%`。
- 买入后现金不低于总资产 `20%`。
- 避免刚卖出后短期重新买入同一标的。
- 买入数量必须符合 A 股 100 股整数倍。
- 如果没有足够高分标的，明确记录 `hold`，不为了交易而交易。

### 3.6 卖出规则

第一版卖出规则：

- `availableShares <= 0` 时禁止卖出，尊重 A 股 T+1。
- 跌破趋势：价格跌破 `MA20` 且策略分数明显下降。
- 风控止损：持仓浮亏或从高点回撤超过阈值。
- 止盈保护：盈利后出现动量衰减，可部分卖出或退出。
- 仓位再平衡：单只股票超过目标仓位上限时减仓。
- 卖出同样必须符合 100 股整数倍。

### 3.7 风控

硬约束：

- 只交易 `sh` / `sz` A 股。
- 每轮最多执行一笔订单。
- 总持仓数量建议不超过 5-8 只。
- 单股仓位不超过 20%。
- 现金保留不低于 20%。
- 不能卖出超过可卖数量。
- 不能在非 A 股交易时段下单。
- 不能在上游市场状态为 closed / not_open 时下单。
- 数据不足时不下单，只记录跳过原因。

## 4. 开发周期

推荐按 4 周推进，先跑稳定闭环，再做增强。

### Phase 0：账号与边界重置（0.5 天）

目标：让新量化账号和旧 AI 账号彻底分开。

任务：

- 注册或接入新的 Signal Arena / Agent World 账号。
- 配置新的 `SIGNAL_ARENA_AGENT_API_KEY` secret。
- 将旧账号从当前 Runner 和公开读取边界移除，不再下单、不再展示。
- 明确页面文案：从 `AI Trader` 改为 `Quant Lab`。
- 当前页面只读取 `quant-v1` 账号数据；旧历史文件即使仍在仓库中，也不参与公开页面、曲线、排名或筛选。

验收：

- Worker 使用新账号读取 `home` 成功。
- 旧账号不会再触发交易。
- 页面只展示新量化账户，不出现旧 AI 账户归档或对照入口。

### Phase 1：数据层与策略基础设施（2-3 天）

目标：建立量化策略需要的数据和计算骨架。

任务：

- 新增 `market-data` 层，封装 `stocks-list`、`stocks`、`stock-history`、`portfolio`、`trades`。
- 增加 D1 表保存历史行情缓存、策略运行快照、因子分数、策略版本。
- 新增指标计算模块，例如 `indicators.ts`。
- 新增策略接口，例如 `StrategyEngine`、`StrategyDecision`、`FactorScore`。
- 为缺失数据、历史长度不足、上游 5xx、限速做明确降级。

验收：

- 给定固定历史行情，可以稳定算出 MA、收益率、波动率、回撤。
- 没有足够历史时返回 `data_insufficient`，不下单。
- 数据和指标测试覆盖核心分支。

### Phase 2：Q-Alpha v1 策略引擎（3-4 天）

目标：用确定性量化规则替代 AI 最终决策。

任务：

- 实现 `Q-Alpha v1` 因子评分。
- 实现候选排序、持仓卖出检查、买入/卖出/观望决策。
- 将 Runner 从 `requestStrictDecision()` 改为调用策略引擎。
- 保留现有 `risk.ts` 作为第二道程序风控。
- 每轮写入完整策略快照：候选分数、触发规则、拒绝原因、最终动作。
- 将 AI provider 从交易闭环中移除，保留给周报任务使用。

验收：

- 同一输入多次运行得到完全一致的交易决策。
- 页面日志能看到“因子分数 + 规则触发 + 风控结果”。
- Worker 测试覆盖买入、卖出、观望、数据不足、T+1、现金不足。

### Phase 3：前端 Quant Lab 改造（2-3 天）

目标：让网站展示“量化策略在做什么”，而不是展示 AI 文本决策。

页面调整：

- `/signal-arena`：策略总览。
  - 策略版本：`Q-Alpha v1`
  - 总资产、收益率、最大回撤、现金比例、持仓数量。
  - 收益曲线和回撤曲线。
  - 当前持仓的策略分、入场理由、风控状态。
  - 最近一轮策略动作。
- `/signal-arena/logs`：策略运行日志。
  - 每轮候选排序。
  - 买入/卖出/观望原因。
  - 因子分数拆解。
  - 风控拦截原因。
  - 点击曲线点弹窗展示该轮完整策略快照。
- `/signal-arena/rank`：竞技排名。
  - 当前量化账号排名。
  - 与排行榜前列的差距。
  - 排名变化趋势。

验收：

- 页面不再把交易结果称为“AI 决策”。
- 任一收益曲线点都能追溯到对应策略快照。
- 首页入口仍保持和其他卡片一致。
- 页面能让用户看懂当前策略版本、主要参数、因子含义、调参风险和下一步观察重点。

### Phase 4：回测与版本化迭代（3-5 天）

目标：让每个策略版本都能用历史数据做离线验证。

任务：

- 增加本地或 Worker 侧回测脚本。
- 回测使用保存的 `stock-history` 数据，不依赖 AI。
- 输出收益率、最大回撤、胜率、盈亏比、换手率、空仓率、持仓数量、交易次数。
- 保存 `strategy_versions`，每个版本有不可变配置。
- 新版本只能通过复制旧版本并修改参数产生，例如 `Q-Alpha v2`。

验收：

- `Q-Alpha v1` 可以跑历史样本回测。
- 回测报告能解释收益来自哪些因子。
- 线上真实运行和回测结果可以并列展示。

### Phase 5：每周 AI 复盘（1-2 天）

目标：让 AI 做研究助理，而不是交易员。

任务：

- 每周收集策略运行、交易、收益、回撤、候选分数。
- 调用 AI 生成中文复盘报告。
- 报告只输出观察、假设和建议实验，不允许输出直接订单。
- 把周报展示在页面中，或存入 D1 作为 `/signal-arena/reports` 后续页面的数据源。

验收：

- 周报能回答：本周赚/亏在哪里、哪些因子有效、哪些规则过严或过松、下周建议验证什么。
- 周报不改变策略参数，不触发交易。
- 周报必须输出“用户可读的调参建议”，格式为：观察现象、可能原因、建议实验、需要验证的数据。

### Phase 6：策略增强循环（持续）

目标：从 `Q-Alpha v1` 进化到更强版本。

候选方向：

- 参数网格搜索：不同 MA、动量窗口、止损阈值、现金比例。
- 市场状态过滤：趋势市、震荡市、下跌市采用不同仓位。
- 多策略组合：趋势策略 + 回撤买入策略 + 防守现金策略。
- 引入外部数据：行业、新闻、财务质量、北向资金等。
- 机器学习：只在数据质量和回测框架稳定后，用于因子权重或市场状态分类。

版本规则：

- v1 上线后不直接改参数。
- 每次改动生成新版本，例如 `Q-Alpha v2`。
- 至少用回测和一段模拟盘结果比较，才决定是否切换主策略。

## 5. 每一版是什么意思

这里的“每一版”指策略版本，不是普通代码版本。

例如：

- `Q-Alpha v1`：第一套固定规则和参数。
- `Q-Alpha v2`：基于回测或周报发现，调整因子权重、卖出规则或风控阈值。
- `Q-Alpha v3`：引入市场状态过滤或多策略组合。
- `Q-Alpha v4`：引入更成熟的研究层，例如外部因子、AI 复盘实验、轻量机器学习或多策略组合评估。

从 v1 到 v2 的触发条件：

- 回测发现某个规则明显拖累收益。
- 真实模拟盘连续出现同类错误。
- 每周 AI 复盘提出的假设被回测验证。
- 收益曲线出现不可接受的回撤或长期空仓。

不能因为某一天没赚钱就随手改参数。量化的核心是保持规则稳定，然后用数据判断规则是否真的有效。

## 6. 策略版本时间表

策略版本不按固定日期强行升级。每一版都要同时满足“最短观察期、样本量、回测证据、风险状态”四类条件，才能进入下一版。时间表只用于规划节奏，真实晋级以证据为准。

### 6.1 总体节奏

| 策略版本 | 最早时间 | 稳妥时间 | 目标 | 晋级后的状态 |
|----------|----------|----------|------|--------------|
| `Q-Alpha v1` | 第 1-2 周上线 | 第 2-3 周稳定 | 建立可运行、可解释、可审计的量化闭环 | 成为第一套主策略 |
| `Q-Alpha v2` | 第 4 周开始 | 第 5-6 周切换 | 基于 v1 的真实运行和回测，优化参数与卖出规则 | 替代 v1 或与 v1 并行观察 |
| `Q-Alpha v3` | 第 7-8 周开始 | 第 9-10 周切换 | 加入市场状态过滤和组合层，让策略适应趋势市/震荡市/下跌市 | 成为更稳的主策略候选 |
| `Q-Alpha v4` | 第 11-12 周开始 | 第 13 周以后 | 引入研究增强层：外部数据、AI 周报实验、多策略评估或轻量 ML | 作为研究增强版，不急着替换 v3 |

### 6.2 Q-Alpha v1：先跑通，不追求花哨

预计开发：

- 工程开发：3-5 个工作日。
- 本地 dry-run / 测试：1-2 个工作日。
- 线上新账号实盘模拟观察：至少 2 个 A 股交易周。

进入 v1 的条件：

- 新账号接入完成，旧 AI 账号不再下单。
- 指标计算、策略评分、风控、下单、日志、页面展示全部可运行。
- 每次决策都能看到因子分数、触发规则和风控结果。
- 连续 3 个交易日没有 Runner 流程级失败。

v1 观察期至少收集：

- 至少 10 个交易日。
- 至少 30 次交易窗口运行记录。
- 至少 3-5 笔真实成交或足够多的明确 `hold` 原因。
- 至少 1 次周报复盘。

v1 的成功标准：

- 策略稳定执行，不因为数据缺失、限速、T+1、100 股规则频繁异常。
- 能解释每一次买入、卖出、观望。
- 最大回撤没有超过预设警戒线，例如 `-8%`。
- 不出现明显的重复追高、刚卖又买、现金耗尽等行为。

如果 v1 亏钱但行为合理，不立刻升级；先观察它是否只是市场环境不利。如果 v1 因规则缺陷反复犯同一种错误，可以提前进入 v2 设计，但不能直接在线上热改 v1。

### 6.3 Q-Alpha v2：优化参数和卖出规则

预计开发：

- 回测与参数实验：2-3 个工作日。
- v2 策略实现：2-3 个工作日。
- v1 / v2 并行影子观察：至少 1 个交易周。

进入 v2 开发的条件：

- v1 已至少运行 10 个交易日，或出现 3 次以上同类可归因错误。
- 已有回测框架，可以比较 v1 与候选 v2。
- 每周复盘或人工观察提出了明确假设，例如“买入阈值太高导致长期空仓”或“止损太慢导致回撤扩大”。

v2 重点改动：

- 调整因子权重，例如趋势/动量/波动惩罚的比例。
- 调整买入阈值，例如 `score >= 70` 是否过严。
- 优化卖出规则，例如止盈保护、趋势跌破、回撤止损。
- 改进持仓数量和现金比例。

v2 晋级为主策略的条件：

- 回测收益、最大回撤、交易次数至少有两项优于 v1。
- 不能通过显著提高交易频率来换取表面收益。
- 影子运行期间，v2 的候选动作能解释并且不违反风控。
- 如果 v2 和 v1 结论冲突，必须记录冲突原因。

v2 不做的事：

- 不引入外部新闻或基本面。
- 不引入机器学习。
- 不做多策略组合。

### 6.4 Q-Alpha v3：加入市场状态和组合层

预计开发：

- 市场状态分类设计：2-3 个工作日。
- 组合层实现：3-4 个工作日。
- v2 / v3 对照观察：至少 2 个交易周。

进入 v3 开发的条件：

- v2 已稳定运行至少 2 个交易周。
- 已经能区分策略在不同市场状态下的表现。
- 回测显示同一套参数在趋势市、震荡市、下跌市表现差异明显。

v3 重点改动：

- 增加市场状态分类：趋势上行、震荡、下跌、防守。
- 不同市场状态使用不同仓位上限和买入阈值。
- 增加组合层：控制行业/风格集中度、持仓数量、现金仓位。
- 增加“不开新仓，只管理风险”的防守模式。

v3 晋级为主策略的条件：

- 最大回撤明显低于 v2。
- 在震荡或下跌环境中减少无效交易。
- 收益可以不显著高于 v2，但风险调整后表现要更好。
- 页面能展示当前市场状态和组合层为什么限制或放行动作。

v3 不做的事：

- 不让 AI 改策略参数。
- 不把 TradingAgents 接入每 15 分钟交易闭环。
- 不用黑箱模型替代规则。

### 6.5 Q-Alpha v4：研究增强版

预计开发：

- 研究方向选择：1 个工作日。
- 数据接入或实验框架：3-5 个工作日。
- 离线评估：至少 2 个交易周或足够历史回测样本。

进入 v4 开发的条件：

- v3 已经稳定运行，且问题不再主要来自基础工程或风控。
- 已有足够日志和回测样本，能评价新增信息是否真的有帮助。
- 至少两期 AI 周报提出同类可验证假设。

v4 候选方向：

- 外部数据因子：行业、新闻、财务质量、资金流、指数状态。
- TradingAgents 风格的多角色周报：技术分析、风险分析、多空辩论、组合经理总结。
- 轻量机器学习：只用于市场状态分类或因子权重建议，不直接输出订单。
- 多策略评估：趋势动量、回撤买入、防守现金策略并行打分。

v4 晋级为主策略或研究层的条件：

- 新增能力能在回测或影子运行中稳定提升风险调整后收益。
- 成本和延迟可控，不影响交易窗口稳定运行。
- 输出能被页面解释，不变成“模型说可以买所以买”。

v4 默认不直接替换 v3。它先作为研究增强层运行，稳定后再决定是否把其中一部分规则固化到 `Q-Alpha v5`。

### 6.6 停止、回滚与冻结规则

任何版本出现以下情况，应冻结升级或回滚到上一版：

- 连续 3 个交易日流程级失败。
- 单日出现 2 次以上可避免的风控拦截。
- 最大回撤超过策略警戒线，例如 `-8%` 预警、`-12%` 强制防守。
- 交易频率异常升高，但收益没有同步改善。
- 页面无法解释某次关键交易的因子和规则来源。

回滚不是失败，而是量化系统的正常保护动作。策略版本的目标不是每天显得更聪明，而是长期可控地变强。

## 7. 用户参与与调参机制

用户参与不是附加文档，而是 Quant Lab 的产品目标之一。这个系统必须让用户知道“策略现在为什么这样交易、效果好不好、我该怎么提出下一次实验”，而不是每次都把判断完全交给开发者。

### 7.1 参与层级

用户参与分四层：

- 每日观察：看收益曲线、回撤、最近动作、Runner 状态、是否有异常失败。
- 每周复盘：看收益、最大回撤、交易次数、胜率、空仓率、因子贡献和本周错误样例。
- 参数实验：提出一个明确假设，例如“买入阈值过高导致长期空仓”，再让系统用回测和影子运行验证。
- 版本审批：决定是否把候选策略从 `v2-candidate` 提升为线上主策略。

### 7.2 使用指南交付物

开发完成时必须交付一份用户可读的使用指南，并随着策略版本更新持续维护。当前草稿路径：

- `docs/signal-arena-quant-lab-user-guide.md`

指南要覆盖：

- 每天应该看哪些指标。
- 每周应该如何判断策略效果。
- 每个关键参数是什么意思。
- 参数调大/调小通常会带来什么影响。
- 什么情况下应该调参，什么情况下应该保持不动。
- 用户应该怎样向 Codex 提出调参请求。

### 7.3 参数调整原则

策略参数不能在当前主版本上静默热改。任何参数调整都走版本化流程：

1. 记录观察：例如“最近 10 个交易日长期空仓”。
2. 提出假设：例如“买入分数阈值 `70` 可能过高”。
3. 新建候选版本：例如 `Q-Alpha v2-candidate-threshold-65`。
4. 回测：比较候选版本和当前主版本的收益、回撤、交易次数、换手率。
5. 影子运行：候选版本只产生日志，不真实下单。
6. 审批切换：只有证据优于当前版本，才切换主策略。

一次实验尽量只改一类参数。不要同时改买入阈值、止损、仓位和因子权重，否则很难知道到底是谁产生了效果。

### 7.4 用户调参请求模板

用户可以直接用这个模板让 Codex 做策略分析：

```text
我观察到：
- 最近 N 天/最近 N 次运行出现了什么现象？

我的假设是：
- 我怀疑哪个参数或规则导致了这个现象？

我想测试：
- 具体希望怎么改？例如买入阈值 70 -> 65，或最大单股仓位 20% -> 15%。

请你：
- 不要直接改线上主策略。
- 先创建候选版本。
- 用最近数据回测或影子运行对比当前版本。
- 输出收益、最大回撤、交易次数、空仓率和结论。
```

如果用户还没有明确假设，也可以这样问：

```text
请你分析最近两周策略日志，找出最值得调参的 1-3 个问题。
不要直接改参数，先解释现象、证据和建议实验。
```

### 7.5 页面需要帮助用户理解的内容

Quant Lab 页面应该给用户足够的信息参与判断：

- 当前策略版本：例如 `Q-Alpha v1`。
- 当前关键参数：买入阈值、卖出阈值、最大仓位、现金保留、止损线、持仓上限。
- 参数解释：每个参数调高/调低通常意味着什么。
- 本轮因子分数：趋势、动量、突破、成交量、波动惩罚、追高惩罚。
- 本轮不交易原因：没有高分候选、现金不足、风控限制、市场关闭、数据不足。
- 版本对比：主版本 vs 候选版本的回测或影子运行结果。
- 周报建议：本周是否建议保持参数、观察更久、或开启候选实验。

第一版不做公开页面上的写入式调参按钮。参数调整先通过版本化代码、测试、回测、影子运行和部署完成。未来如果需要，可以再做带鉴权的私有 Admin 页面。

## 8. TradingAgents 调研

### 8.1 它是什么

GitHub 上最相关的项目是 [TauricResearch/TradingAgents](https://github.com/TauricResearch/TradingAgents)。它是一个 Python 多智能体 LLM 金融交易研究框架，官方描述为 `Multi-Agents LLM Financial Trading Framework`。

它的核心结构是模拟真实交易团队：

- 基本面分析师。
- 情绪分析师。
- 新闻分析师。
- 技术分析师。
- 多空研究员辩论。
- Trader Agent 汇总报告并形成交易建议。
- 风控团队和 Portfolio Manager 负责最终评估。

它基于 LangGraph，支持多个 LLM provider，并提供 CLI / Python package 用法。官方 README 也明确强调这是研究用途，交易表现会受模型、温度、数据和非确定性因素影响。

另一个相关项目是 [hsliuping/TradingAgents-CN](https://github.com/hsliuping/TradingAgents-CN)。它是中文增强版，提供 A 股支持、中文界面、FastAPI/Vue/MongoDB/Redis 等增强能力。但该项目采用混合许可证，`app/` 和 `frontend/` 属于需要商业授权的专有部分，不适合直接复制进本站。

### 8.2 对我们的作用

有用，但不适合第一版直接接入交易闭环。

适合借鉴：

- 角色拆分：分析、研究、交易、风控、组合管理分层。
- 风控前置：交易建议不能绕过组合和风险检查。
- 记忆与复盘：保存决策日志，再用真实结果反思。
- 多空辩论：作为周报的一部分，帮助发现策略盲点。

不适合直接采用的原因：

- 它是 LLM 多智能体框架，而我们当前目标是确定性量化策略。
- LLM 决策有非确定性，不利于回测和复现。
- 它偏 Python/LangGraph，而本站 Runner 当前在 Cloudflare Worker TypeScript 环境。
- 接入成本、模型成本和延迟都高于第一版需要。
- 中文增强版的前后端目录存在专有许可限制。

建议结论：

- `Q-Alpha v1` 不直接引入 TradingAgents。
- Phase 5 以后可以做一个独立研究任务：每周把策略日志喂给 TradingAgents 风格的多角色复盘流程。
- 如果未来要实验它，应该作为离线研究服务或 GitHub Actions / 服务器任务，不应该在交易时段每 15 分钟实时调用。

## 9. 页面数据清单

量化版页面应该新增或替换这些字段：

- 策略版本：`Q-Alpha v1`、启用时间、配置摘要。
- 参数说明：买入阈值、卖出阈值、仓位上限、现金保留、止损线、冷却期等用户可理解字段。
- 账户表现：总资产、收益率、最大回撤、现金比例、持仓数量。
- 收益曲线：继续使用等距点位，保留真实时间 tooltip。
- 回撤曲线：展示当前策略是否在扩大亏损。
- 因子排名：本轮候选股 top N，每个候选的趋势、动量、突破、波动、成交量、总分。
- 持仓策略分：每个持仓当前分数、入场分数、是否触发卖出观察。
- 决策日志：本轮最终动作、规则命中、风控结果、订单结果。
- 版本对比：当前版本 vs 历史版本的收益、回撤、交易次数。
- 周报：AI 复盘摘要、下周假设、待回测实验。
- 用户指南入口：解释如何阅读页面和如何提出调参请求。

已有曲线点弹窗可以保留，但弹窗内容改为：

- 决策前账户状态。
- 候选排名。
- 最终买/卖/观望规则。
- 风控检查。
- 订单结果。
- 该次动作之后的收益点。

## 10. 数据拉取策略

不建议每轮拉取所有股票的全部历史数据。

推荐：

- 每日开盘前：刷新 A 股标的池。
- 每个交易窗口：读取账户、持仓、近期交易、top movers。
- 每个交易窗口：对持仓股和初筛候选股拉取 `stock-history`。
- 历史行情缓存到 D1/KV，已有数据增量更新。
- 如果触发限速，当前轮降级为“只管理持仓风险，不开新仓”。

这样能兼顾数据新鲜度和上游 rate limit。

## 11. 风险与注意事项

- 这仍然是模拟交易学习项目，不是投资建议。
- 第一版不能承诺赚钱，但可以承诺可复现、可解释、可回测。
- 数据源如果没有复权价格，长期回测会受分红、拆股等因素影响。
- Signal Arena 订单每 15 分钟结算，策略不能假设立即成交。
- A 股 T+1 和 100 股限制必须继续由程序风控兜底。
- 公开页面不得展示 API key、上游认证 header、模型 key、完整私有响应。
- 旧 AI 账号历史不进入当前公开体验；新量化账号曲线、日志和排名只读取当前 `quant-v1` 数据。
- 参数调整不能绕过回测、影子运行和版本审批。

## 12. 下一步

建议按顺序执行：

1. 部署前执行 D1 migration 与 Worker secret 切换，确保 `SIGNAL_ARENA_AGENT_API_KEY` 指向新的量化账号。
2. 先用 `dryRun=true` 手动运行，确认 `home / portfolio / stocks-list / stock-history` 都能返回。
3. 在 A 股交易时段执行一次非 dry-run，确认新 run 写入 `account_scope=quant-v1` 与 `strategy_trace_json`。
4. 确认线上 `/signal-arena`、`/signal-arena/logs`、`/signal-arena/rank` 只显示新量化账号数据。
5. 后续进入回测、候选版本和每周 AI 复盘阶段，继续维护 `docs/signal-arena-quant-lab-user-guide.md`。
